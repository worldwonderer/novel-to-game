# 需求 intake：先框定产品，再动手

小说转游戏最贵的错误都发生在写一行代码之前——**平台选错、类型选错、画风选错、
尺度选错、核心幻想没定**。这些决定一旦被下游各阶段各自默认，做到一半才暴露，返工代价
极大。intake gate 的职责是把这套**产品框架**变成一次显式、用户确认、之后不可静默改写的
事实（`PRODUCT_BRIEF.md`），供拆解、概念、设计、美术、构建全程遵守。

这一步不是走过场。**用户第一次给出小说时，必须先过这道门，才进拆解。**

## 怎么问：给默认值，不做审讯

不要甩一串空问题给用户。正确姿势是**先读懂原作、替用户填好一份推荐草案，再请他确认或
改**：

1. 速读原作足够判断：语言、题材、基调、时代/设定、有无成人/暴力/情爱内容、情绪内核。
2. 由此为每个维度**推出一个有依据的推荐默认值**（语言→市场→对标；题材→类型+画风；
   基调→分级）。参考 [intake-benchmark-reference.md](intake-benchmark-reference.md)。
3. 把整份草案摆给用户，请他确认或逐项改。离散的高杠杆选择（平台、画风、分级/NSFW、
   类型/对标方向）用 AskUserQuestion 提，推荐项放第一并标（推荐）；核心幻想一句话用自由
   文本。一次问清关键几项，别追问到底。
4. 用户不在场时（AFK/自动流程）：按默认值推进，但在 `PRODUCT_BRIEF.md` 里把每条标为
   **未确认假设**并醒目列出，让后续任何人一眼能挑战。

## 必定维度（intake 清单）

| # | 维度 | 要定什么 | 默认从哪来 | 不定的代价 |
|---|---|---|---|---|
| 1 | 平台与形态 | 客户端(PC/主机) / 网页 / 小程序 / 移动App；竖屏或横屏；主设备 | 体量与题材：重叙事重养成→端/App；即开即玩/社交裂变→网页/小程序 | 决定 UI、控制、时长、体量、画风上限 |
| 2 | 目标市场与界面语言 | 主市场、首发界面语言 | **小说语言→对应市场**（硬规则，见下）；界面语言默认=市场语言 | 决定对标、类型惯例、内容敏感点、文化符号 |
| 3 | 游戏类型 + 对标名作 | 主类型/子类型 + 2-4 款该市场玩家真正熟知的对标 | 题材→类型常见映射 + 参考表 | 单一最强锚点；不定则概念阶段全靠猜 |
| 4 | 美术画风 | 像素 / 2D手绘 / 立绘+背景 / 2.5D / 卡通渲染3D / 写实3D / 水墨工笔… | 由平台 × 题材 × 体量推 | 决定辨识度、制作可行性、招牌画面 |
| 5 | 内容分级 / NSFW | 全年龄 / 12+ / 16+ / 成人向(暗示) / 成人向(露骨 NSFW)；若露骨，定**热度天花板** | 由原作尺度 + 市场 + 平台合规推 | 定错则文案与美术全线返工 |
| 6 | 核心幻想 / 情绪卖点 | 一句话：玩家扮谁、要什么、该在哪"爽/怕/心动" | 由原作情绪内核推 | 不定则做出"系统正确却没人想玩"的东西 |
| 7 | 单局时长与结构 | 垂直切片时长；单局 vs 长线；有无养成/日常循环 | 默认 10-30 分钟可完整游玩切片 | 决定循环密度与范围 |
| 8 | 发行/商业化定位（可选但常影响玩法） | 买断/独立发行 vs 免费+内购/抽卡/体力 | 由平台 + 市场推 | 决定是否需要 gacha、日常、资源循环 |

## 两条硬规则

**语言→市场→对标**：小说是什么语言，就对标那个市场里玩家真正熟悉的名作——中文小说
对标中国市场爆款，英文对标西方，日文对标日本，韩文对标韩国。对标是为了借它**证明过的
具体玩法/画风/商业原则**，每个对标都要说清"借它哪一条"，不是拿名字装饰，也不能拿另一
市场的成功直接证明本地接受度。映射与候选见
[intake-benchmark-reference.md](intake-benchmark-reference.md)。

**尺度与平台互相约束**：分级不替用户默认最保守，也不替用户放开——**必须显式确认**。若
用户要 NSFW/露骨，要当场点破它对平台与发行的约束：小程序与多数应用商店禁成人内容，
露骨内容通常只能走网页/PC/特定成人平台。分级定了要连**露骨档的热度天花板**一起定
（暗示 / 点到肌肤即止 / 解衣肌肤气息但视角克制 / 更直白），避免"放开了"却没说放到哪。
无论分级多高，**不涉及真实人物、不性化任何未成年角色**是不可逾越的底线，与用户偏好无关。

## 产出：PRODUCT_BRIEF.md

一页锁定这八个维度，外加：对标矩阵（每个对标借哪一条）、明确非目标、合规注意、以及所有
**未确认假设**清单。它进入"上游事实不得下游静默改写"的保护范围（见
[pipeline-contract.md](pipeline-contract.md)）：概念阶段的产品定义直接**继承**它而非重猜，
美术方向读它的画风与平台，构建读它的平台形态，任何阶段要改产品框架都要回到这里显式改，
不许下游各自替换成"更好用的版本"。
