# 构建执行技法

从实际把一本小说做成可玩网页游戏中沉淀的四条可复用技法。都是原则，不是工具教程；
用什么模型、框架、命令由实现环境决定。

## 1. 灰盒先行 + 皮肤层（资产可替换）

先用色块/形状/字形把完整核心循环跑通，验证规则与范围，**不等美术**。所有可视元素
经一个**键查表**取材：每个键可选带 `sprite`/贴图，未加载则回退到灰盒。美术产出后按键
放入即替换，**不改引擎与交互逻辑**。好处：玩法与美术解耦，可分别并行推进与验证，
缺图也永远能跑。角色立绘、地砖、战斗背景全部走这层。

## 2. 可复现随机（种子 RNG）

即使玩法需要概率（命中、暴击、相克浮动、逃跑），也让**一切随机走一个可设种子的
RNG**。自测断言"同种子 + 同操作 = 同结果"，覆盖行动顺序、伤害序列、胜负、状态转换、
养成结算。这样概率手感与可回归的确定性测试并存——既有"回合制该有的波动"，又能
逐字节比对回归。无种子的随机会让自检失去意义。

## 3. 把实现模型当"导演对象"驱动

当前会话不亲自写全部代码时，把**已批准的设计**整份交给强编码模型/CLI，让它自主决定
架构与文件拆分；美术资产交给图像模型。迭代靠**真实运行 + 截图**：起服务、走核心路径、
截图看构图/可读性/遮挡、修控制台与运行错误，再重复。要求实现方自带 node 级与浏览器级
自测（可复现、0 报错、存证截图），把"我改了什么、测试结果"作为完成证据，而非口头声称。

## 4. 多视角试玩闭环

有了可玩版本后，做一轮**独立多视角评审**：资深策划（系统深度/平衡/节奏）、资深同类玩家
（手感/是否像该品类）、新手（上手/可读性）、美术（构图/一致性/细节）、剧情（叙事/忠于原作）。
每个视角只交回**少数确切可实施**的问题（一句话问题 + 严重度 + 具体改法），拒绝含糊或大厂式
无法落地的建议。汇总去重后驱动**一轮**打磨再回归验证。这一步专门拦截"看似能玩、实则每回合
无决策/翻盘技打空/BOSS 是搓血"这类只有玩过才暴露的缺陷。
