# 游戏世界设计方法

## 先做能被试玩证伪的设计

把每条设计写成可被试玩证伪的假设，并尽早做出一小段粗糙但完整的可玩内容——关卡、
玩家行为和反馈回路要和设计决策放在一起看，而不是等内容量堆满再验证。每条体验支柱
都必须有一段最小可玩证据，先解决最危险的玩家体验问题，再写内容量和包装。

不要把试玩做成“喜欢吗”的问卷。观察玩家第一次做什么、卡在哪、忽略什么、是否能从
反馈修正下一次行动；提问只用来解释行为原因。

## 对标只解决冲突

对标的价值是提供已经验证的选择原则。例如“形态必须改变可达区域”可以否决只换模型
的变身，“伪装允许进入但错误行为会暴露”可以否决无条件通行。不要照抄对标的功能
数量，也不要让支持类型吞掉主类型。

设计文档要说明核心体验的优先顺序。时间、资产或实现发生冲突时，先删除外围战斗、
收集和演出，再保护最能证明原作转化的玩家动作。

## 类型最低契约

只写会影响玩家决策的类型规则。回合制通常需要信息公开程度、行动经济、结算顺序和
随机性；实时动作通常需要输入、时机、空间和恢复；管理游戏通常需要时间推进、资源
流与反馈延迟。不要把其中一套固定成所有游戏的模板。

## 从行动到世界响应

用一个最小因果模型描述关键规则：

```text
当前状态 + 玩家选择 + 世界规则 -> 新状态 + 可读反馈
```

核心循环必须说明优秀玩家会注意或做出什么不同。没有技巧差异、取舍或变化语境的
循环通常只是交互。

用两层循环防止“单句核心循环”变成口号：

```text
5-30 秒：观察/选择 -> 输入 -> 即时反馈 -> 状态变化
1-5 分钟：组合多个短循环 -> 新压力或新信息 -> 改变策略 -> 阶段结果
```

两层都要写普通玩家和熟练玩家的不同；如果所有玩家都只需重复同一安全动作，系统还没有
形成玩法。

核心动词的反馈必须在同一拍内可读：输入 → 画面/声音 → 状态变化闭合，让玩家立即看懂
“发生了什么”。反馈缺失、滞后，或只靠数字弹字，按缺陷处理，不当作手感。

## 系统预算

第一版原型使用一个核心系统、最多两个支持系统和一个标志性场面。支持系统只有在增加
核心决策时才存在。库存、制作、技能树、天气和开放世界不能仅为显得完整而加入。

只为核心规则写清输入、选择、后果、反馈和可被滥用的策略。模型已经知道如何把规则
实现成状态和代码。

验收分两层：自动测试证明确定性规则、状态和产物契约；无提示试玩证明玩家能否从
反馈理解因果。不要用搜索策划文案中的固定句子代替任何一层。

## 关卡节奏

用最短闭环证明概念，并按“教会 → 变式 → 组合 → 检验”安排关卡：

1. 清楚的目标和第一次行动；
2. 安全反馈让玩家理解核心动词（教会）；
3. 改变空间、节奏、敌方意图或资源，使机械重复失效（变式）；
4. 让两个已知规则产生原作特有的组合或标志性场面（组合）；
5. 在较少提示下检验玩家能否迁移规则，并给出可理解的结果、失败与重试理由（检验）。

为每个空间说明镜头可见范围、主路线、可选风险路线和地标。高压后需要恢复段；失败
必须显示发生了什么以及玩家下次能改变什么。优先通过空间、反应和后果传递设定。
