# 质量验证契约

## 必须证明

- 页面能够构建、启动并持续渲染，无关键错误或资源失败；
- 真实输入导致玩家或世界产生可观察变化；
- 核心循环能够推进；
- 设计要求的成功、失败或其他结果可到达；
- 重开返回有效初始状态；
- 每个目标视口保持可读且可操作。
- 策划要求的每种界面语言均无缺字、截断、错序或关键术语漂移。

系统规则的验证用状态和结果断言；内容理解与可读性用无提示试玩和真实画面。测试文件
中出现某句策划文案，只能证明文字没变，不能证明规则正确或游戏可玩。

对画布游戏至少比较加载后与输入/时间推进后的真实帧，排除空白、零尺寸、遮挡和只有
背景动画的假运行。截图检查玩家、目标、威胁、界面、反馈和招牌画面是否符合
`ART_DIRECTION.md`。

## 自检不能自证

用与实现**同一套规则、同一份常量、同一张词表**写成的自检，只能证明实现自洽，不能
证明它做对了：过滤逻辑跑通，用同一份规则去验收就必然通过，无论漏掉多少。验收断言
必须来自独立来源——另行编写的检查清单、从策划文档反向推出的期望值，或与生成逻辑
无关的第三方扫描。

同理，实现方（外部模型或 CLI）自带的测试套件不能是唯一的担保人。它按自己的理解写
断言，理解错了断言也跟着错。关键不变量要另写一份直接对照 `GAME_DESIGN.md` 的检查，
与实现方的套件分开跑。

## 取舍要用极端策略验证

设计承诺"玩家要在 A 与 B 之间取舍"时，把引擎从界面剥离出来直接跑完整局：**纯 A、
纯 B、混合**各跑一遍固定种子，比较最终状态与结局。

- 某条单一策略通吃 → 取舍不存在，玩家没有真正做过选择；
- 两条极端走到同一结果 → 选择没有后果；
- 作为"诱饵"的可见指标（排行榜、评分、进度条）若在放弃它的打法下也不下滑 → 诱惑
  不成立，后续任何"反转/醒悟"都失去前提。

同一榜单里玩家与 NPC 必须用**同一量纲**计分；一边取两项之和、另一边取单项，名次
会系统性失真，诱饵随之失效。

这类缺陷读代码和单点断言都看不出来，只有跑完整局才暴露。

## 严重度

- `blocker`：无法构建/启动、主画面空白、核心输入无效、崩溃或无法完成/重开。
- `major`：关键机制、状态、画面可读性或目标视口失败，但仍有替代路径。
- `minor`：不阻断体验的局部视觉、反馈或警告问题。

保留足以复现结论的命令、状态、日志和截图，不要求为了数量保存重复证据。
