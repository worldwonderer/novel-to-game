# 质量验证契约

## 必须证明

- 页面能够构建、启动并持续渲染，无关键错误或资源失败；
- 真实输入导致玩家或世界产生可观察变化；
- 核心循环能够推进；
- 设计要求的成功、失败或其他结果可到达；
- 重开返回有效初始状态；
- 每个目标视口保持可读且可操作。
- 策划要求的每种界面语言均无缺字、截断、错序或关键术语漂移。

系统规则的验证用状态和结果断言；内容理解与可读性用无提示试玩和真实画面。测试文件
中出现某句策划文案，只能证明文字没变，不能证明规则正确或游戏可玩。

对画布游戏至少比较加载后与输入/时间推进后的真实帧，排除空白、零尺寸、遮挡和只有
背景动画的假运行。截图检查玩家、目标、威胁、界面、反馈和招牌画面是否符合
`ART_DIRECTION.md`。

## 自检不能自证

用与实现**同一套规则、同一份常量、同一张词表**写成的自检，只能证明实现自洽，不能
证明它做对了：过滤逻辑跑通，用同一份规则去验收就必然通过，无论漏掉多少。验收断言
必须来自独立来源——另行编写的检查清单、从策划文档反向推出的期望值，或与生成逻辑
无关的第三方扫描。

同理，实现方（外部模型或 CLI）自带的测试套件不能是唯一的担保人。它按自己的理解写
断言，理解错了断言也跟着错。关键不变量要另写一份直接对照 `GAME_DESIGN.md` 的检查，
与实现方的套件分开跑。

## 取舍要用极端策略验证

设计承诺"玩家要在 A 与 B 之间取舍"时，把引擎从界面剥离出来直接跑完整局：**纯 A、
纯 B、混合**各跑一遍固定种子，比较最终状态与结局。

- 某条单一策略通吃 → 取舍不存在，玩家没有真正做过选择；
- 两条极端走到同一结果 → 选择没有后果；
- 作为"诱饵"的可见指标（排行榜、评分、进度条）若在放弃它的打法下也不下滑 → 诱惑
  不成立，后续任何"反转/醒悟"都失去前提。

同一榜单里玩家与 NPC 必须用**同一量纲**计分；一边取两项之和、另一边取单项，名次
会系统性失真，诱饵随之失效。

这类缺陷读代码和单点断言都看不出来，只有跑完整局才暴露。

## 核心幻想与上手必须验证，不只验证"能跑通"

规则全绿、能通关、能重开，不等于游戏交付了它承诺的体验。玩家的真实抱怨往往是"点进去
不知道做什么""没感受到卖点"，而这些在纯机制断言下全部通过。**趣味无法客观证明，但
"承诺的体验有没有出现"可以观察**——照下面查，不打主观分。

**首次上手（用没读过策划的眼睛走一遍）**：

- 从标题进入后，玩家能否在设计规定的时间内（通常 ≤60 秒、首个有意义动作 ≤30 秒）做出
  第一个真正的动作，并当场得到反馈？卡在"这一屏我该点哪"就是 `major`。
- 一个没读过 `GAME_DESIGN.md` 的人，玩前两分钟后能否说出：我是谁、我要什么、我现在在
  往哪使劲？说不出核心目标，是 `major`——不是"细节没懂"，是主线卖点没传达。
- 引导是否用后果教而不是甩术语词典；核心动作是否一上来就被教会。

**核心幻想是否被"演"出来**：

- 逐条对照 `CONCEPT.md` / `GAME_DESIGN.md` 的**体验支柱与它们的"可观察试玩证据"**：每条
  证据在真实游玩里到底出现没有。承诺"争宠/被追逐/步步高升/绝地翻盘"这类卖点，是变成了
  有画面有反馈的**场面**，还是只剩一个变动的数字？只有数字、没有戏，就是卖点没交付。
- 逐条检查设计写明的**"会否决该支柱的失败现象"**有没有发生（例如"每回合其实没有真正的
  决策""诱饵指标放弃了也不掉"）。发生即按其严重度记。

这一步专门拦截"系统正确、却没人想玩/不知道怎么玩"的缺陷，只有当作新玩家真正玩过才暴露。
它验证的是**设计承诺有没有兑现**，不是"设计本身好不好玩"——若证据都出现了体验仍平淡，
那是要回上游改设计的发现，如实报告，不在 QA 这层假装通过。

- `blocker`：无法构建/启动、主画面空白、核心输入无效、崩溃或无法完成/重开。
- `major`：关键机制、状态、画面可读性或目标视口失败，但仍有替代路径。
- `minor`：不阻断体验的局部视觉、反馈或警告问题。

保留足以复现结论的命令、状态、日志和截图，不要求为了数量保存重复证据。
